```go
func TestUserController_Get(t *testing.T) {
  tester := gonest.MustNewTestApp(UserModule, func(b *gonest.TestBuilder) {
    gonest.MustOverride[IUserService](b, &UserServiceMock{ /* ... */ })
  })
  defer tester.Close()

  res := tester.MustRequest(gonest.HttpGet, "/users/42", nil)
  res.AssertStatus(t, gonest.HttpStatusOk)
  res.AssertJsonPath(t, "id", int64(42))
}
```

## `MustNewTestApp(module, fn)`

Ejecuta el **mismo** bootstrap de tres fases que `MustNewApp`, con la excepción de que no inicia un
listener HTTP real — las rutas se registran para su despacho en memoria.

## `MustOverride[T](builder, mockValue)`

Sustituye el constructor real de un provider por un mock, codificado por el `reflect.Type` de `T`.

<Callout type="warn">
  `T` **debe ser una interfaz**. Go no puede cambiar el comportamiento de un `*struct` concreto
  en tiempo de ejecución — no hay vtable para redirigir. Los servicios que están destinados a ser simulados (mocked)
  en las pruebas deben ser inyectados/expuestos por interfaz, no por puntero concreto.
</Callout>

## Despachando solicitudes

`tester.MustRequest(method, path, body)` despacha una solicitud en memoria — sin
puerto de red real — contra la aplicación de prueba. `body` está codificado en JSON si no es nil.

Atajos de verbos HTTP raíz utilizados en pruebas/despacho:
`gonest.HttpMethod`, `HttpGet`, `HttpPost`, `HttpPut`, `HttpDelete`, `HttpQuery`.

## Asserções (Assertions)

`TestResponse.AssertStatus(t, want)` y `AssertJsonPath(t, path, want)`
(notación de punto, ej: `"id"`, `"address.zip"`) fallan la prueba con un mensaje
claro en caso de discordancia.

<Callout type="info">
  Consulta [`.examples`](https://github.com/gonest-dev/gonest/tree/main/.examples)
  en el repositorio gonest tanto para un ejemplo mínimo como uno completo más denso ejerciendo
  estos auxiliares de prueba.
</Callout>
