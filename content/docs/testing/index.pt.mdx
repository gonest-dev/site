```go
func TestUserController_Get(t *testing.T) {
  tester := gonest.MustNewTestApp(UserModule, func(b *gonest.TestBuilder) {
    gonest.MustOverride[IUserService](b, &UserServiceMock{ /* ... */ })
  })
  defer tester.Close()

  res := tester.MustRequest(gonest.HttpGet, "/users/42", nil)
  res.AssertStatus(t, gonest.HttpStatusOk)
  res.AssertJsonPath(t, "id", int64(42))
}
```

## `MustNewTestApp(module, fn)`

Executa o **mesmo** bootstrap de três fases que `MustNewApp`, com exceção de não iniciar um
listener HTTP real — as rotas são registradas para despacho em memória.

## `MustOverride[T](builder, mockValue)`

Substitui o construtor real de um provider por um mock, usando o `reflect.Type` de `T` como chave.

<Callout type="warn">
  `T` **deve ser uma interface**. O Go não pode trocar o comportamento de um `*struct` concreto
  em tempo de execução — não há vtable para redirecionar. Os serviços que se destinam a ser mockados
  em testes devem ser injetados/expostos por interface, não por ponteiro concreto.
</Callout>

## Despachando requisições

`tester.MustRequest(method, path, body)` despacha uma requisição em memória — sem
porta de rede real — contra o app de teste. `body` é codificado em JSON se não for nil.

Atalhos de verbos HTTP na raiz usados em testes/despacho:
`gonest.HttpMethod`, `HttpGet`, `HttpPost`, `HttpPut`, `HttpDelete`, `HttpQuery`.

## Asserções

`TestResponse.AssertStatus(t, want)` e `AssertJsonPath(t, path, want)`
(notação de ponto, ex: `"id"`, `"address.zip"`) falham o teste com uma mensagem
clara na divergência.

  esses auxiliares (helpers) de teste.
</Callout>
