Um `Module` é a unidade que o gonest inicializa. Construa um com
`gonest.NewModule(func(module *gonest.Module) {...})`.

```go
var UserModule = gonest.NewModule(func(module *gonest.Module) {
  module.Providers(UserProvider)
  module.Controllers(UserController)
})
```

## Métodos

| Método | Descrição |
|---|---|
| `Providers(...)` | Registra Providers de propriedade deste Module. |
| `Controllers(...)` | Registra Controllers de propriedade deste Module. |
| `Imports(...)` | Importa outros Modules, obtendo acesso aos seus Providers exportados. |
| `Exports(...)` | Expõe Providers para Modules que importam este. |
| `Use(...)` | Registra Middleware global. **Apenas no Module raiz.** |
| `Filters(...)` | Registra Filters de exceção globais. **Apenas no Module raiz.** |
| `Listeners(...)` | Registra ouvintes de eventos para o [Emitter](/docs/emitter). |
| `Schedulers(...)` | Registra tarefas do [Scheduler](/docs/scheduler). |
| `Name("...")` | Define um nome legível usado em mensagens de erro. |

<Callout type="warn">
  `Use` e `Filters` são apenas para a raiz quando aplicados no nível do Module — eles
  se aplicam globalmente a todas as requisições. `Use`/`Filters` por Controller também existem
  (veja [Request Pipeline](/docs/request-pipeline)) e são combinados com os globais.
</Callout>

## Aplicando Middleware e Filters globais

```go
var AppModule = gonest.NewModule(func(module *gonest.Module) {
  module.Use(RequestIdMiddleware)
  module.Filters(FooFilter)
})
```

## Dynamic Modules

O NestJS precisa de um tipo especial `DynamicModule` e da convenção `forRoot()`/`register()` porque
`@Module()` é um **decorator** — sua metadata fica fixa em tempo de definição da classe, então injetar
config de runtime nela precisa de um escape hatch. gonest não tem essa limitação: `Module` já é um
valor Go comum construído por uma função builder, então um "dynamic module" é só uma função normal que
fecha sobre `options` e retorna um `*gonest.Module` — sem tipo especial, sem suporte de framework
nenhum:

```go
type DatabaseOptions struct {
  DSN string
}

func DatabaseModuleForRoot(opts DatabaseOptions) *gonest.Module {
  connProvider := gonest.NewProvider(func(provider *gonest.Provider) {
    provider.Constructor(func() *Connection { return connect(opts.DSN) })
  })
  return gonest.NewModule(func(module *gonest.Module) {
    module.Providers(connProvider)
    module.Exports(connProvider)
  })
}
```

<Callout type="warn">
  Construa o dynamic module UMA VEZ numa `var` de pacote, mesma convenção que todo módulo estático do
  gonest já usa — `Module.Imports` deduplica por identidade de ponteiro Go, então duas chamadas
  SEPARADAS de `DatabaseModuleForRoot(mesmasOpts)` produzem dois valores `*Module` DIFERENTES (duas
  conexões reais, não uma compartilhada). Importe a MESMA variável em todo lugar que precisar:

  ```go
  var DatabaseModule = DatabaseModuleForRoot(DatabaseOptions{DSN: os.Getenv("DATABASE_URL")})

  var AppModule = gonest.NewModule(func(module *gonest.Module) {
    module.Imports(DatabaseModule) // mesmo *Module onde quer que seja importado
  })
  ```
</Callout>

Não existe equivalente ao `forRootAsync` do Nest (config resolvida a partir de um serviço injetado) —
a função builder de um `Module` não tem capacidade de `MustInject` própria (a montagem de módulos roda
ANTES de qualquer `Provider` ser resolvido). Esse caso nem precisa de um, na verdade: o Provider que
precisa da config resolve ela via `MustInject` dentro do próprio `Constructor`, o mesmo padrão de
dependência Provider-a-Provider que o gonest já suporta em todo lugar — veja [Config
Loading](/docs/config) pra um exemplo real de um Provider se construindo a partir de uma dependência
em tempo de `Constructor`.

## Lazy loading não tem equivalente em gonest — de propósito, não esquecimento

O `LazyModuleLoader` do NestJS adia o trabalho real de um módulo pro primeiro uso via `import()`
dinâmico do JavaScript — código que ainda não foi baixado/parseado é carregado sob demanda, o que
importa pra ambientes sensíveis a cold-start (serverless, webworkers) que querem pular caminhos de
código que uma invocação específica nunca toca.

<Callout type="warn">
  **Isso não tem equivalente idiomático em Go, e não é uma lacuna que o gonest pretende preencher.**
  Um binário Go é compilado antecipadamente — o código de TODO pacote importado já está DENTRO do
  binário, sempre, não importa se uma chamada `Module.Imports` específica roda ou não. Não existe
  estado "ainda não baixado" pra código Go adiar (a stdlib tem o pacote `plugin`, mas é notoriamente
  frágil, só Linux/macOS, e não é usado em lugar nenhum deste projeto nem do ecossistema Go em geral).
  O problema que o `LazyModuleLoader` resolve simplesmente não existe pra um binário compilado — o
  custo que o Nest adia pro tempo do primeiro request, o Go já pagou inteiro em tempo de `go build`.
</Callout>

O paralelo REAL mais próximo — adiar o `Constructor` caro de um Provider até o primeiro uso de
verdade, em vez de eager durante `NewApp` (todo Provider `Singleton` registrado resolve concorrente no
bootstrap hoje, use alguém ou não) — é uma ideia DIFERENTE, não implementada, não é "lazy loading" no
sentido de carregamento de módulo. Veja `INSIGHT-LAZY.md` na raiz do repo pro raciocínio completo,
incluindo um cenário concreto (um binário único estilo Lambda atendendo múltiplos tipos de evento,
onde só a árvore de Provider de UM tipo é realmente necessária por invocação).
