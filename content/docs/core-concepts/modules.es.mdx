Un `Module` es la unidad que gonest inicializa. Construye uno con
`gonest.NewModule(func(module *gonest.Module) {...})`.

```go
var UserModule = gonest.NewModule(func(module *gonest.Module) {
  module.Providers(UserProvider)
  module.Controllers(UserController)
})
```

## Métodos

| Método | Descripción |
|---|---|
| `Providers(...)` | Registra Providers propiedad de este Module. |
| `Controllers(...)` | Registra Controllers propiedad de este Module. |
| `Imports(...)` | Importa otros Modules, obteniendo acceso a sus Providers exportados. |
| `Exports(...)` | Expone Providers a Modules que importan este. |
| `Lazy(func(l *gonest.LazyModule) {...})` | Decide qué Module(s) `Imports`/`Exports` a partir de un valor de config resuelto desde dentro del grafo de DI — mira [Decidiendo un import desde dentro del grafo de DI](#decidiendo-un-import-desde-dentro-del-grafo-de-di-modulelazy) abajo. |
| `Use(...)` | Registra Middleware global. **Solo en el Module raíz.** |
| `Filters(...)` | Registra Filters de excepción globales. **Solo en el Module raíz.** |
| `Listeners(...)` | Registra escuchas de eventos para el [Emitter](/docs/emitter). |
| `Schedulers(...)` | Registra trabajos del [Scheduler](/docs/scheduler). |
| `Name("...")` | Establece un nombre legible utilizado en mensajes de error. |

<Callout type="warn">
  `Use` y `Filters` son solo para la raíz cuando se aplican a nivel de Module — se
  aplican globalmente a todas las solicitudes. `Use`/`Filters` por Controller también existen
  (consulta [Request Pipeline](/docs/request-pipeline)) y se combinan con los globales.
</Callout>

Un Provider exportado por su tipo **concreto** solo es inyectable por ese tipo concreto. Para permitir
que un Module que importa este otro haga `MustInject` de un Provider por una **interfaz** que
implementa, envuélvelo con `gonest.ProviderAs[T](ref)` antes de pasarlo a `Providers`/`Exports` —
consulta [Exportando un Provider como una interfaz](/docs/core-concepts/providers#exportando-un-provider-como-una-interfaz).

## Aplicando Middleware y Filters globales

```go
var AppModule = gonest.NewModule(func(module *gonest.Module) {
  module.Use(RequestIdMiddleware)
  module.Filters(FooFilter)
})
```

## Dynamic Modules

NestJS necesita un tipo especial `DynamicModule` y la convención `forRoot()`/`register()` porque
`@Module()` es un **decorator** — su metadata queda fija en tiempo de definición de la clase, así que
inyectar config de runtime en ella necesita una vía de escape. gonest no tiene esa limitación: `Module`
ya es un valor Go común construido por una función builder, así que un "dynamic module" es solo una
función normal que cierra sobre `options` y retorna un `*gonest.Module` — sin tipo especial, sin
soporte de framework alguno:

```go
type DatabaseOptions struct {
  DSN string
}

func DatabaseModuleForRoot(opts DatabaseOptions) *gonest.Module {
  connProvider := gonest.NewProvider(func(provider *gonest.Provider) {
    provider.Constructor(func() *Connection { return connect(opts.DSN) })
  })
  return gonest.NewModule(func(module *gonest.Module) {
    module.Providers(connProvider)
    module.Exports(connProvider)
  })
}
```

<Callout type="warn">
  Construye el dynamic module UNA VEZ en una `var` de paquete, la misma convención que ya usa todo
  módulo estático de gonest — `Module.Imports` deduplica por identidad de puntero Go, así que dos
  llamadas SEPARADAS a `DatabaseModuleForRoot(mismasOpts)` producen dos valores `*Module` DIFERENTES
  (dos conexiones reales, no una compartida). Importa la MISMA variable en todo lugar que la
  necesites:

  ```go
  var DatabaseModule = DatabaseModuleForRoot(DatabaseOptions{DSN: os.Getenv("DATABASE_URL")})

  var AppModule = gonest.NewModule(func(module *gonest.Module) {
    module.Imports(DatabaseModule) // mismo *Module donde sea que se importe
  })
  ```
</Callout>

### Decidiendo un import desde DENTRO del grafo de DI: `Module.Lazy`

`DatabaseModuleForRoot` arriba cubre options que el LLAMADOR pasa explícitamente. El otro caso que
cubre el `forRootAsync` de Nest — un Module eligiendo qué Module(s) hermano(s) importar/exportar a
partir de un valor de config resuelto desde un servicio INYECTADO, no desde un argumento pasado desde
afuera — tiene un equivalente real en gonest: `Module.Lazy(func(l *gonest.LazyModule) {...})`. Dentro
de ese callback, `gonest.MustInject[T](l)` construye de forma eager y síncrona un Provider ya
registrado en los propios `Providers(...)` de ESTE Module, y `l.Imports(...)`/`l.Exports(...)` se
comportan exactamente como `Module.Imports`/`Module.Exports`:

```go
var Config_ = gonest.NewProvider(func(p *gonest.Provider) {
  p.Constructor(func() *Config {
    return gonest.MustParse[Config](gonest.Dotenv(), configSchema) // env:"NOTIFICATION_DRIVER"
  })
})

var Module_ = gonest.NewModule(func(m *gonest.Module) {
  m.Providers(Config_)
  m.Exports(Config_)
  m.Lazy(func(l *gonest.LazyModule) {
    config := gonest.MustInject[*Config](l)
    switch config.Driver {
    case "sms":
      l.Imports(sms.Module_)
      l.Exports(sms.Module_)
    default:
      l.Imports(email.Module_)
      l.Exports(email.Module_)
    }
  })
})
```

Este es el `.examples/notification-driver` real, funcionando — `NOTIFICATION_DRIVER` elige
`email.Module_` o `sms.Module_` en el bootstrap, decidido desde dentro del grafo de DI en vez de una
función libre llamada desde `main()`.

<Callout type="warn">
  El `MustInject[T](l)` de `Module.Lazy` es intencionalmente restringido — solo ve Providers
  registrados en los propios `Providers(...)` de ESTE Module, antes en la misma función builder (no
  los Providers de un Module importado, ya que `Imports` todavía no está finalizado — eso es
  justamente lo que `Lazy` está decidiendo). El Provider objetivo también debe ser **autocontenido**:
  su `Constructor` no puede llamar `MustInject` (sin dependencias anidadas), y debe ser
  `ScopeSingleton` (el valor por defecto). Ambos causan panic en el bootstrap si no — `Lazy` es para un
  Provider con forma de `Config` leyendo `.env`/flags, no un resolvedor síncrono de subgrafo genérico.
</Callout>

## Lazy loading no tiene equivalente en gonest — a propósito, no por olvido

<Callout type="info">
  No confundir con `Module.Lazy` arriba — ese elige QUÉ Module importar, de forma síncrona, en el
  bootstrap. Esta sección trata de otra cosa: el `LazyModuleLoader` de NestJS, que difiere la carga del
  CÓDIGO de un módulo hasta el primer uso.
</Callout>

`LazyModuleLoader` de NestJS difiere el trabajo real de un módulo hasta el primer uso vía `import()`
dinámico de JavaScript — código que aún no fue descargado/parseado se carga bajo demanda, lo cual
importa para entornos sensibles a cold-start (serverless, webworkers) que quieren saltarse rutas de
código que una invocación específica nunca toca.

<Callout type="warn">
  **Esto no tiene equivalente idiomático en Go, y no es una carencia que gonest planee llenar.** Un
  binario Go se compila por adelantado — el código de TODO paquete importado ya está DENTRO del
  binario, siempre, sin importar si una llamada `Module.Imports` específica corre o no. No existe un
  estado "aún no descargado" para que el código Go difiera (la stdlib tiene el paquete `plugin`, pero
  es notoriamente frágil, solo Linux/macOS, y no se usa en ningún lugar de este proyecto ni del
  ecosistema Go en general). El problema que `LazyModuleLoader` resuelve simplemente no existe para
  un binario compilado — el costo que Nest difiere hasta el primer request, Go ya lo pagó entero en
  tiempo de `go build`.
</Callout>

El paralelo REAL más cercano — diferir el `Constructor` caro de un Provider hasta el primer uso real,
en vez de eager durante `NewApp` (todo Provider `Singleton` registrado se resuelve concurrente en el
bootstrap hoy, lo use alguien o no) — es una idea DIFERENTE, no implementada, no es "lazy loading" en
el sentido de carga de módulo. Mira `INSIGHT-LAZY.md` en la raíz del repo para el razonamiento
completo, incluyendo un escenario concreto (un binario único estilo Lambda atendiendo múltiples tipos
de evento, donde solo el árbol de Provider de UN tipo es realmente necesario por invocación).
