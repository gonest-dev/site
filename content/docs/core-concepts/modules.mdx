A `Module` is the unit gonest bootstraps. Build one with
`gonest.NewModule(func(module *gonest.Module) {...})`.

```go
var UserModule = gonest.NewModule(func(module *gonest.Module) {
  module.Providers(UserProvider)
  module.Controllers(UserController)
})
```

## Methods

| Method | Description |
|---|---|
| `Providers(...)` | Register providers owned by this module. |
| `Controllers(...)` | Register controllers owned by this module. |
| `Imports(...)` | Import other modules, gaining access to their exported providers. |
| `Exports(...)` | Expose providers, or re-export a whole imported `*Module`, to modules that import this one — see below. |
| `Lazy(func(l *gonest.LazyModule) {...})` | Decide which module(s) to `Imports`/`Exports` from a config value resolved inside the DI graph — see [Deciding an import from inside the DI graph](#deciding-an-import-from-inside-the-di-graph-modulelazy) below. |
| `Use(...)` | Register global middleware. **Root module only.** |
| `Filters(...)` | Register global exception filters. **Root module only.** |
| `Listeners(...)` | Register event listeners for the [Emitter](/docs/emitter). |
| `Schedulers(...)` | Register [Scheduler](/docs/scheduler) jobs. |
| `Name("...")` | Set a readable name used in error messages. |

<Callout type="warn">
  `Use` and `Filters` are root-only when applied at the module level — they
  apply globally to every request. Per-controller `Use`/`Filters` also exist
  (see [Request Pipeline](/docs/request-pipeline)) and combine with the
  global ones.
</Callout>

## Applying global middleware and filters

```go
var AppModule = gonest.NewModule(func(module *gonest.Module) {
  module.Use(RequestIdMiddleware)
  module.Filters(FooFilter)
})
```

## Re-exporting a whole module

`Exports` takes both individual providers and whole imported `*Module`
values in the same call — mirrors NestJS's `exports: [SomeProvider,
CommonModule]`, which accepts providers and modules in one array. Passing a
`*Module` re-exports it: to make an ENTIRE imported module's own exports
visible transitively, import it and pass it to `Exports` alongside (or
instead of) individual providers:

```go
var DatabaseModule = gonest.NewModule(func(module *gonest.Module) {
  module.Providers(daoProviders...) // 20+ DAO providers
  module.Exports(daoProviders...)
})

var InfraModule = gonest.NewModule(func(module *gonest.Module) {
  module.Imports(DatabaseModule)
  module.Exports(DatabaseModule) // re-export: a *Module, not a provider
})

var AppModule = gonest.NewModule(func(module *gonest.Module) {
  module.Imports(InfraModule) // never imports DatabaseModule directly
  module.Controllers(SomeController) // can still MustInject any DAO
})
```

This chains to any depth (a module can re-export a module that itself
re-exports another), only propagates what the re-exported module ALREADY
exports (never bypasses its own encapsulation), and a re-exported module
must also be passed to `Imports` on the same module — `Exports`ing
something you didn't import is a Stage 1 assembly error, not a silent
no-op.

<Callout type="info">
  `Exports` accepts anything implementing `gonest.ExportableRef` — a
  `ProviderRef` (individual provider) or a `*Module` (whole re-export), in
  any mix, in one call.
</Callout>

A provider exported by its **concrete** type is only injectable by that
concrete type. To let an importing module `MustInject` a provider by an
**interface** it implements, wrap it with `gonest.ProviderAs[T](ref)` before
passing it to `Providers`/`Exports` — see [Exporting a provider as an
interface](/docs/core-concepts/providers#exporting-a-provider-as-an-interface).

## Dynamic modules

NestJS needs a special `DynamicModule` return type and `forRoot()`/`register()` convention because
`@Module()` is a **decorator** — its metadata is fixed at class-definition time, so injecting runtime
config into it needs an escape hatch. gonest has no such constraint: `Module` is already a plain Go
value built by a builder function, so a "dynamic module" is just a regular function that closes over
options and returns a `*gonest.Module` — no special type, no framework support needed:

```go
type DatabaseOptions struct {
  DSN string
}

func DatabaseModuleForRoot(opts DatabaseOptions) *gonest.Module {
  connProvider := gonest.NewProvider(func(provider *gonest.Provider) {
    provider.Constructor(func() *Connection { return connect(opts.DSN) })
  })
  return gonest.NewModule(func(module *gonest.Module) {
    module.Providers(connProvider)
    module.Exports(connProvider)
  })
}
```

<Callout type="warn">
  Build the dynamic module ONCE into a package-level `var`, the same convention every static module in
  gonest already uses — `Module.Imports` deduplicates by Go pointer identity, so two SEPARATE calls to
  `DatabaseModuleForRoot(sameOpts)` produce two DIFFERENT `*Module` values (two real connections, not
  one shared). Import the same variable everywhere you need it:

  ```go
  var DatabaseModule = DatabaseModuleForRoot(DatabaseOptions{DSN: os.Getenv("DATABASE_URL")})

  var AppModule = gonest.NewModule(func(module *gonest.Module) {
    module.Imports(DatabaseModule) // same *Module wherever it's imported
  })
  ```
</Callout>

### Deciding an import from inside the DI graph: `Module.Lazy`

`DatabaseModuleForRoot` above covers options the CALLER supplies explicitly. The other case Nest's
`forRootAsync` covers — a module picking which sibling module(s) to import/export based on a config
value resolved from an INJECTED service, not from an argument passed in from outside — has a real
gonest equivalent: `Module.Lazy(func(l *gonest.LazyModule) {...})`. Inside that callback,
`gonest.MustInject[T](l)` eagerly and synchronously constructs a provider already registered via this
SAME module's own `Providers(...)`, and `l.Imports(...)`/`l.Exports(...)` behave exactly like
`Module.Imports`/`Module.Exports`:

```go
var Config_ = gonest.NewProvider(func(p *gonest.Provider) {
  p.Constructor(func() *Config {
    return gonest.MustParse[Config](gonest.Dotenv(), configSchema) // env:"NOTIFICATION_DRIVER"
  })
})

var Module_ = gonest.NewModule(func(m *gonest.Module) {
  m.Providers(Config_)
  m.Exports(Config_)
  m.Lazy(func(l *gonest.LazyModule) {
    config := gonest.MustInject[*Config](l)
    switch config.Driver {
    case "sms":
      l.Imports(sms.Module_)
      l.Exports(sms.Module_)
    default:
      l.Imports(email.Module_)
      l.Exports(email.Module_)
    }
  })
})
```

This is the real, working `.examples/notification-driver` — `NOTIFICATION_DRIVER` picks `email.Module_`
or `sms.Module_` at bootstrap, decided from inside the DI graph instead of a free function called from
`main()`.

<Callout type="warn">
  `Module.Lazy`'s `MustInject[T](l)` is intentionally narrow — it only sees providers registered via
  THIS module's own `Providers(...)`, earlier in the same builder function (not an imported module's
  providers, since `Imports` isn't finalized yet — that's what `Lazy` is deciding). The targeted
  provider must also be **self-contained**: its `Constructor` cannot itself call `MustInject` (no
  nested dependencies), and it must be `ScopeSingleton` (the default). Both panic at bootstrap
  otherwise — `Lazy` is for a `Config`-shaped provider reading `.env`/flags, not a general synchronous
  sub-graph resolver.
</Callout>

## Lazy loading has no gonest equivalent — by design, not oversight

<Callout type="info">
  Not to be confused with `Module.Lazy` above — that picks WHICH module to import, synchronously, at
  bootstrap. This section is about something else entirely: NestJS's `LazyModuleLoader`, which defers
  loading a module's CODE until first use.
</Callout>

NestJS's `LazyModuleLoader` defers a module's real work to first use via JavaScript's dynamic
`import()` — code that hasn't been downloaded/parsed yet gets loaded on demand, which matters for
cold-start-sensitive environments (serverless, webworkers) that want to skip loading code paths a
given invocation never touches.

<Callout type="warn">
  **This has no idiomatic Go equivalent, and it's not a gap gonest plans to fill.** A Go binary is
  compiled ahead of time — every imported package's code is already IN the binary, always, whether or
  not a given `Module.Imports` call ever runs. There is no "not yet downloaded" state for Go code to
  defer (the stdlib's `plugin` package exists but is notoriously fragile, Linux/macOS-only, and not
  used anywhere in this project or the wider Go ecosystem). The problem `LazyModuleLoader` solves
  simply doesn't exist for a compiled binary — the cost NestJS defers to first-request time, Go already
  paid in full at `go build` time.
</Callout>

The closest REAL parallel — deferring an expensive `Provider`'s `Constructor` until first actual use,
rather than eagerly at `NewApp` (every registered `Singleton` provider resolves concurrently during
bootstrap today, whether or not anything ever calls it) — is a different, unshipped idea, not
"lazy loading" in the module-loading sense. See `INSIGHT-LAZY.md` in the repo root for the full
reasoning, including a concrete scenario (a single Lambda-style binary handling multiple event types,
where only one event type's Provider tree is actually needed per invocation).
