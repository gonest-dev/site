---
title: Providers y Alcances de DI
description: Cómo gonest construye y define el alcance de las instancias de dependencias
---

Un `Provider` construye una instancia para el grafo de DI. Construye uno con
`gonest.NewProvider(func(provider *gonest.Provider) {...})`.

```go
var UserProvider_ = gonest.NewProvider(func(provider *gonest.Provider) {
  provider.Scope(gonest.ScopeSingleton)
  provider.Constructor(func() *UserService { return &UserService{} })
})
```

## Convención de nombres

Todo var a nivel de paquete exportado producido por un builder de gonest
(`NewProvider`, `NewController`, `NewModule`, `NewListener`, `NewScheduler`,
`NewResolver`, y [`ProviderAs`](#exportando-un-provider-como-una-interfaz)
abajo) recibe un guion bajo al final — `UserProvider_`,
`NotificationController_` — incondicionalmente, no solo cuando existe una
colisión real con un struct/type del mismo nombre. Una sola regla, en vez de
un juicio caso por caso.

## Firmas de constructor

`Constructor` acepta exactamente una de estas cuatro formas de función — **nunca** una
función con parámetros de dependencia:

- `func() T`
- `func() (T, error)`
- `func(ctx) T`
- `func(ctx) (T, error)`

Las dependencias se resuelven con `MustInject`/`MustInjectAll` **dentro** del
constructor (builder), antes de que se llame a `Constructor` — no como parámetros para la
función del constructor misma.

## Alcances (Scopes)

| Alcance | Comportamiento |
|---|---|
| `gonest.ScopeSingleton` | Una instancia para toda la aplicación (por defecto). |
| `gonest.ScopeTransient` | Una nueva instancia en cada resolución. |
| `gonest.ScopeRequest` | Una instancia por solicitud HTTP. |

## Resolviendo dependencias

`MustInject[T](owner)` resuelve por tipo y da panic a menos que se encuentre exactamente una coincidencia.
Tanto los tipos de puntero (`*UserService`) como los tipos de interfaz requieren un registro **exacto**
— ya no existe una alternativa estructural. Un Provider solo se vuelve resoluble por una interfaz `T`
si fue envuelto explícitamente para `T` vía
[`ProviderAs`](#exportando-un-provider-como-una-interfaz), abajo.

`MustInjectAll[T](owner)` devuelve cada coincidencia para `T` — utilizado para
patrones de plugin/estrategia — y nunca da panic en 0 o N coincidencias.

```go
type Connectable interface{ Ping() bool }

var SystemController_ = gonest.NewController(func(controller *gonest.Controller) {
  connectables := gonest.MustInjectAll[Connectable](controller)
  _ = connectables
})
```

## Exportando un Provider como una interfaz

El `Constructor` de un `Provider` registra un tipo **concreto** (`*Service`, no la interfaz que
casualmente implementa). `MustInject[T]`/`MustInjectAll[T]` solo coinciden con una interfaz `T` cuando
algún Provider fue envuelto explícitamente para `T` vía `gonest.ProviderAs[T](ref)` — la alternativa
estructural anterior (`reflect.Type.Implements()`, que coincidía con cualquier Provider que satisficiera
`T`) fue eliminada por completo. Un Provider concreto que nunca se envuelve no es resoluble por ninguna
interfaz que casualmente satisfaga, aunque sea el único implementador en el alcance.

Este es el patrón de backend intercambiable (swappable-backend) — un `Notifier` enviado vía `email` o
`sms`, un repositorio `Person` respaldado por `memory` o `postgres` — donde el código consumidor debe
depender solo de la interfaz, nunca de qué adaptador concreto está realmente conectado.
[`.examples/notification-driver`](https://github.com/gonest-dev/gonest/tree/main/.examples/notification-driver)
es un ejemplo real y ejecutable; aquí está su adaptador `email`:

```go
// notifier/impl/email/service.go
type Service struct{}

func (s *Service) Send(to, message string) error {
  fmt.Printf("[email] to=%s message=%q\n", to, message)
  return nil
}

var Provider_ = gonest.NewProvider(func(provider *gonest.Provider) {
  provider.Constructor(func() *Service { return &Service{} })
})
```

```go
// notifier/impl/email/module.go
// AsNotifier_ envuelve Provider_, registrándolo explícitamente como port.Notifier.
var AsNotifier_ = gonest.ProviderAs[port.Notifier](Provider_)

var Module_ = gonest.NewModule(func(module *gonest.Module) {
  module.Providers(Provider_, AsNotifier_) // el registro concreto de Provider_
  module.Exports(AsNotifier_)              // sigue siendo obligatorio
})
```

El código consumidor inyecta solo la interfaz, sin saber qué adaptador responde por ella:

```go
notify := gonest.MustInject[port.Notifier](controller)
```

<Callout type="warn">
  `ProviderAs[T](ref)` nunca dirige la construcción por sí mismo — el registro concreto de `ref`
  TAMBIÉN debe pasarse a `Providers(...)`, como se muestra arriba. Envolver un Provider sin también
  registrar su ref concreto falla en voz alta en el bootstrap de `NewApp`/`MustNewApp`, no en la primera
  llamada a `MustInject[T]`. `T` debe ser un tipo de interfaz (`ProviderAs` da panic inmediatamente si
  no), y envolver el resultado de otra llamada a `ProviderAs` (encadenamiento) también da panic —
  envuelve el `ProviderRef` concreto original de nuevo para cada interfaz que necesite exponer.
</Callout>
