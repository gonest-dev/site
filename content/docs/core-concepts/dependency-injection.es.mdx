gonest resuelve las dependencias por **tipo**, no por nombre o token de cadena.

## `MustInject[T](owner)`

Resuelve exactamente una instancia de `T`. Da panic si se encuentran cero o más de una
coincidencia.

- **Tipos de puntero** (`*UserService`) requieren una coincidencia exacta de tipo.
- **Tipos de interfaz** coinciden con un registro exacto, o recurren a
  `reflect.Type.Implements()` si exactamente un Provider satisface la interfaz.

```go
userService := gonest.MustInject[*UserService](controller)
```

## `MustInjectAll[T](owner)`

Resuelve **cada** Provider que coincide con `T`. Nunca da panic en 0 o N coincidencias — úsalo
para patrones de plugin/estrategia (plugin/strategy patterns) donde múltiples Providers implementan legítimamente
la misma interfaz.

```go
type Pingable interface {
  Name() string
  Ping(ctx context.Context) error
}

var HealthController = gonest.NewController(func(controller *gonest.Controller) {
  controller.Path("/health")
  pingables := gonest.MustInjectAll[Pingable](controller)
  _ = pingables
})
```

<Callout type="info">
  Este es el mismo patrón utilizado para [Health Checks](/docs/health-checks): cualquier
  número de servicios puede implementar `Pingable`, y el Controller de salud los
  descubre todos automáticamente.
</Callout>

## Dónde se puede llamar a `MustInject`

`MustInject`/`MustInjectAll` son legales dentro de los constructores de `Provider`, `Controller`
— en cualquier lugar donde el framework garantice que la Fase 1 (resolución de Providers) ya
se ha completado. Consulta [Bootstrap](/docs/getting-started/bootstrap).

Los constructores de `Guard` son la única excepción: **no** tienen soporte para `MustInject`,
porque un solo Guard se puede adjuntar a través de múltiples Controllers/Modules —
no hay un único "propietario" (owner) con el cual resolver.
