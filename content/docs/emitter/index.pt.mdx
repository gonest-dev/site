---
title: Event Emitter
description: Typed, fire-and-forget events between providers
---

O `gonest.Emitter` é um **singleton do framework** — sempre resolvível via
`MustInject[*gonest.Emitter]` em qualquer módulo, com zero registro explícito.

```go
type UserCreatedEvent struct{ UserID int64 }

var UserCreatedListener = gonest.NewListener(func(listener *gonest.Listener) {
  gonest.MustOn[UserCreatedEvent](listener, func(ctx context.Context, event UserCreatedEvent) {
    // handle the event
  })
})

var UserModule = gonest.NewModule(func(module *gonest.Module) {
  module.Listeners(UserCreatedListener)
})

var UserProvider = gonest.NewProvider(func(provider *gonest.Provider) {
  emitter := gonest.MustInject[*gonest.Emitter](provider)
  provider.Constructor(func() *UserService { return &UserService{emitter: emitter} })
})
```

<Callout type="info">
  `MustOn[EventType](listener, handler)` é uma **função** livre, não um
  método — o Go não permite parâmetros de tipo (type parameters) em métodos.
</Callout>

## Semântica do Emit

`Emitter.Emit(event)` é **fire-and-forget** (dispare e esqueça): uma goroutine por ouvinte (listener) registrado
para aquele tipo exato de evento, retorna imediatamente, nunca bloqueia quem
chamou (caller). O panic ou erro de um ouvinte é recuperado internamente e nunca
se propaga.

**Não há correspondência de eventos por padrão de string/wildcard** — os eventos são tipados por
struct do Go.

## Subscriptions dinâmicas (`Subscribe[T]`)

`MustOn` registra um handler estático, com a vida toda do app. `gonest.Subscribe[T](emitter, done)` é
seu par dinâmico — um canal que vive só enquanto `done` continua aberto (tipicamente a vida de uma
conexão), fechado automaticamente assim que dispara. É sobre isso que as Subscriptions de GraphQL
são construídas (ver [GraphQL](/docs/graphql)):

```go
func(ctx *gonest.GraphqlContext, emit func(any)) {
  for event := range gonest.Subscribe[OrderCreatedEvent](emitter, ctx.Done()) {
    emit(event)
  }
}
```
