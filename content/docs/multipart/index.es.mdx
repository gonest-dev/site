---
title: Streaming de Formularios Multipart
description: Carga de archivos con streaming real, sin buffering
---

La mayoría de los frameworks (incluido el propio `multer`/`FileInterceptor` de NestJS) hace buffer de un
archivo subido (upload) en memoria o disco antes de que se ejecute el código del handler. En cambio, gonest
hace streaming: `onFile` se dispara en el instante en que se ve una parte de un archivo en el flujo (stream) multipart
en bruto, permitiendo al handler reenviar bytes directamente a S3 (por ejemplo) sin llegar a
hacer un buffer del archivo completo.

```go
type CreatePostForm struct {
  Title string `form:"title"`
}

var createPostFormSchema = gonest.NewSchema[CreatePostForm](func(t *CreatePostForm, m *gonest.Schema) {
  m.Property(&t.Title).String().Required()
})

var PostController = gonest.NewController(func(controller *gonest.Controller) {
  controller.RoutePost("/", func(route *gonest.Route) {
    route.FormBody(createPostFormSchema, "file") // documents multipart/form-data in OpenAPI
    route.Handler(func(req *gonest.Request, res *gonest.Response) {
      form := gonest.MustParse[CreatePostForm](req.Body().Form(func(f *gonest.FormFile) error {
        return uploadToS3(f.Filename(), f.ContentType(), f.Reader())
      }), createPostFormSchema)
      res.Json(map[string]string{"title": form.Title})
    })
  })
})

func main() {
  app := gonest.MustNewApp[gonest.FiberApp](AppModule, gonest.AppOptions{EnableFormStreaming: true})
  app.MustListen(":3000")
}
```

## `FormFile`

El callback pasado a `req.Body().Form(...)` recibe un `*gonest.FormFile`
con `Filename()`, `ContentType()`, y `Reader()`.

## `AppOptions.EnableFormStreaming`

Esta flag se aplica a **toda la aplicación**, no por ruta — alterna `StreamRequestBody` + `DisablePreParseMultipartForm` de Fiber juntos. Las rutas
JSON/param/query existentes no se ven afectadas.

Nueva familia de etiquetas: `form:"..."`. La misma vía de escape [`Custom(fn)`](/docs/validation/custom-validators)
funciona aquí también, recibiendo el valor de la cadena en bruto.

<Callout type="warn">
  Trade-off, no es un bug: si `onFile` devuelve un error no nulo, el recorrido se detiene
  y produce una `*BadRequestException` incluyendo el nombre del campo y el propio
  mensaje de error del callback. Pero si un campo de formulario **posterior** resulta ser
  inválido, `onFile` puede **ya haberse disparado** para los archivos vistos anteriormente en el
  flujo — diseña tu propio rollback (ej: eliminar un objeto de S3 parcialmente subido)
  para el escenario "carga iniciada, pero la solicitud resultó inválida".
</Callout>
