---
title: Streaming de Formulários Multipart
description: Upload de arquivos com streaming de verdade, sem buffering
---

A maioria dos frameworks (incluindo o próprio `multer`/`FileInterceptor` do NestJS) faz buffer de um
arquivo enviado (upload) para a memória ou disco antes que o código do handler seja executado. O gonest, por outro lado, faz streaming: `onFile` é disparado no instante em que uma parte do arquivo é vista no fluxo (stream) multipart
bruto, permitindo que o handler encaminhe bytes diretamente para o S3 (por exemplo) sem nunca
fazer buffer do arquivo inteiro.

```go
type CreatePostForm struct {
  Title string `form:"title"`
}

var createPostFormSchema = gonest.NewSchema[CreatePostForm](func(t *CreatePostForm, m *gonest.Schema) {
  m.Property(&t.Title).String().Required()
})

var PostController = gonest.NewController(func(controller *gonest.Controller) {
  controller.RoutePost("/", func(route *gonest.Route) {
    route.FormBody(createPostFormSchema, "file") // documents multipart/form-data in OpenAPI
    route.Handler(func(req *gonest.Request, res *gonest.Response) {
      form := gonest.MustParse[CreatePostForm](req.Body().Form(func(f *gonest.FormFile) error {
        return uploadToS3(f.Filename(), f.ContentType(), f.Reader())
      }), createPostFormSchema)
      res.Json(map[string]string{"title": form.Title})
    })
  })
})

func main() {
  app := gonest.MustNewApp[gonest.FiberApp](AppModule, gonest.AppOptions{EnableFormStreaming: true})
  app.MustListen(":3000")
}
```

## `FormFile`

O callback passado para `req.Body().Form(...)` recebe um `*gonest.FormFile`
com `Filename()`, `ContentType()`, e `Reader()`.

## `AppOptions.EnableFormStreaming`

Esta flag se aplica a **todo o aplicativo**, não por rota — ela ativa/desativa o
`StreamRequestBody` + `DisablePreParseMultipartForm` do Fiber em conjunto. Rotas
JSON/param/query existentes não são afetadas.

Nova família de tags: `form:"..."`. A mesma saída de escape [`Custom(fn)`](/docs/validation/custom-validators)
funciona aqui também, recebendo o valor da string bruta.

<Callout type="warn">
  Trade-off, não é um bug: se `onFile` retornar um erro não-nulo, a travessia (walk) para
  e produz uma `*BadRequestException` incluindo o nome do campo e a
  própria mensagem de erro do callback. Mas se um campo de formulário **posterior** se revelar
  inválido, o `onFile` pode **já ter disparado** para arquivos vistos anteriormente no
  stream — planeje seu próprio rollback (ex: excluir um objeto S3 parcialmente enviado)
  para o cenário "upload iniciado, mas a requisição se revelou inválida."
</Callout>
