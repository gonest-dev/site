`GraphqlResolver` is analogous to `Controller` — registered via `module.Resolvers(...)`, reusing the
same `Schema`/`Parse[T]`/`MustInject` REST already uses. No duplicate validation or DI between REST
and GraphQL.

```go
package ex

import "gonest.dev/gonest"

var UserResolver = gonest.NewGraphqlResolver(func(resolver *gonest.GraphqlResolver) {
  userService := gonest.MustInject[*UserService](resolver)

  resolver.Query("user", func(query *gonest.GraphqlQuery) {
    query.Args(userIdParamsSchema)   // same Schema already used for REST validation
    query.Returns(userEntitySchema)  // same Schema drives the generated SDL too

    query.Handler(func(ctx *gonest.GraphqlContext) any {
      args := gonest.MustParse[UserIdParams](ctx.Args(), userIdParamsSchema)
      return userService.Get(args.UserId) // the returned value IS the data
    })
  })
})
```

<Callout type="info">
  `MustInject`'s owner must be `resolver` itself (or another Resolver/Controller) — the
  `Query`/`Mutation`/`Subscription` builders passed to their own callbacks have no DI resolution
  capability of their own.
</Callout>

## Query and Mutation

Both share the exact same shape — `.Args(schema)`, `.Returns(schema)`, `.Handler(func(ctx *gonest.GraphqlContext) any)`.
The Handler's return value directly becomes the GraphQL response's `data`, same as NestJS — there is
no separate `Response`/write-side (GraphQL has no status codes or headers to justify one):

```go
resolver.Mutation("createUser", func(mutation *gonest.GraphqlMutation) {
  mutation.Args(userEntitySchema)
  mutation.Returns(userEntitySchema)

  mutation.Handler(func(ctx *gonest.GraphqlContext) any {
    input := gonest.MustParse[UserEntity](ctx.Args(), userEntitySchema)
    return userService.Create(input)
  })
})
```

`gonest.GraphqlMutation` is a type alias of `gonest.GraphqlQuery` — the GraphQL root type a field ends
up under (`Query` vs `Mutation` in the generated SDL) is decided by which method registered it, not
by a distinct Go type.

Both dispatch through one fixed HTTP endpoint, `POST /graphql`, following the standard
GraphQL-over-HTTP shape (`{query, variables, operationName}` → `{data, errors}`).

## Subscription

Fundamentally different from Query/Mutation — a STREAM, not request-response. The resolver stays
alive emitting N values while the client remains connected, so its `Handler` has a different
signature (`func(ctx, emit)`, not `func(ctx) any`):

```go
resolver.Subscription("orderCreated", func(subscription *gonest.GraphqlSubscription) {
  subscription.Args(orderFilterSchema)
  subscription.Returns(orderEntitySchema)

  subscription.Handler(func(ctx *gonest.GraphqlContext, emit func(any)) {
    args := gonest.MustParse[OrderFilter](ctx.Args(), orderFilterSchema)
    emitter := gonest.MustInject[*gonest.Emitter](resolver)

    for event := range gonest.Subscribe[OrderCreatedEvent](emitter, ctx.Done()) {
      if event.CustomerId == args.CustomerId {
        emit(event)
      }
    }
  })
})
```

`gonest.Subscribe[T](emitter, done)` is `Emitter`'s dynamic counterpart to `MustOn`/`Emit` — a channel
that lives only as long as `done` stays open (typically one connection's lifetime), closed
automatically once the client disconnects. `orderService.Create(...)` keeps calling
`emitter.Emit(OrderCreatedEvent{...})` exactly as it already would for a static `MustOn` listener —
the Subscription doesn't know or care who publishes the event, only its type.

### Transport: SSE and WebSocket

Subscriptions connect through two endpoints, registered automatically the moment at least one
`Subscription` exists — `:name` selects which one to stream, arguments arrive via a single
`?args=<JSON object>` query parameter (there is no request body on a long-lived connection):

```
GET /graphql/stream/onOrderCreated?args={"customerId":42}   -- SSE
GET /graphql/ws/onOrderCreated?args={"customerId":42}       -- WebSocket
```

Both call `Subscription.Handler` directly — neither goes through `graphql-go/graphql`'s own execution
engine, which never shipped a subscription streaming engine of its own.

<Callout type="warn">
  A panic inside a Subscription `Handler` is recovered (it won't crash the process), but there is no
  GraphQL-shaped error frame sent to the client yet — a recognized gap, not resolved in this version.
</Callout>

## Automatic SDL generation

The generated SDL comes from the exact same `Schema` REST already uses — no second, parallel type
declaration to keep in sync.

### Format branches become Custom Scalars

Every format branch (`Email`/`Uuid`/`Uri`/`Hostname`/`Ipv4`/`Ipv6`/`Password`/`Byte`/`Binary`/`DateTime`/`Date`)
maps to a GraphQL Custom Scalar — GraphQL's 5 built-in scalars (`Int`/`Float`/`String`/`Boolean`/`ID`)
have no concept of "this is an email":

```go
type UserEntity struct {
  Id        int64     `json:"id"`
  Email     string    `json:"email"`
  CreatedAt time.Time `json:"createdAt"`
}

var userEntitySchema = gonest.NewSchema(func(t *UserEntity, s *gonest.Schema) {
  s.Property(&t.Id).Integer().Required()
  s.Property(&t.Email).Email().Required()
  s.Property(&t.CreatedAt).DateTime().Required()
})
```

Generates:

```graphql
scalar Email
scalar DateTime

type UserEntity {
  id: Int!
  email: Email!
  createdAt: DateTime!
}
```

Every distinct format used anywhere in the app deduplicates to ONE `scalar` declaration, never
repeated per field.

### Fully custom types (e.g. MongoDB's `primitive.ObjectID`)

A type with no OpenAPI `format` equivalent (already handled for REST via `Custom(fn)`) has no obvious
scalar name. `Custom(fn)` gains `.GraphqlScalar(name)`, relevant only when the `Schema` is consumed by
a Resolver — REST/OpenAPI ignore this value entirely:

```go
m.Property(&t.Id).Custom(decodeObjectID).GraphqlScalar("ObjectID").Required()
```

Two fields sharing the same `.GraphqlScalar("ObjectID")` deduplicate to one `scalar ObjectID`
declaration. `Custom(fn)` without `.GraphqlScalar(name)` on a schema consumed by a Resolver fails at
**SDL build time** (a configuration error, not a per-request failure).
